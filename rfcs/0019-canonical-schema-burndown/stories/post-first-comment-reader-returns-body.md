## Context

`Post#first_comment` in Rails is a reader override, not just the association:

```ruby
# vendor/rails/activerecord/test/models/post.rb:56-59
def first_comment
  super.body
end
has_one :first_comment, -> { order("id ASC") }, class_name: "Comment"
```

The public `first_comment` reader returns the comment's **body string**
(`super` invokes the association reader → `Comment`, then `.body`), NOT the
`Comment` object.

trails' `Post` (`packages/activerecord/src/test-helpers/models/post.ts`,
`hasOne("firstComment", { ... })` around line 267) only declares the
association — there is no `firstComment` reader override — so
`post.firstComment` resolves to the `Comment` record. The materialized declare
`firstComment: Comment | null` faithfully reflects trails' CURRENT behavior;
the divergence is the missing reader override, which the declare-materialization
generator cannot synthesize (it has no view of Ruby method overrides).

Surfaced by Codex review of PR #4193.

## Acceptance criteria

- [ ] `Post` implements the Rails `first_comment` reader override so the public
      reader returns the first comment's `body` (string), matching
      `post.rb:56-59`. Decide the trails-idiomatic shape given association reads
      are async (`await post.firstComment`) — likely a distinct method, not a
      clobber of the association proxy.
- [ ] The corresponding `declare firstComment` surface reflects the override's
      return type (string), regenerated by the materializer (or hand-declared if
      the override is a method the generator does not model).
- [ ] Any consumer in `*.test.ts` relying on `post.firstComment` returning a
      `Comment` is converged to the Rails behavior.
- [ ] Test names match Rails verbatim.
