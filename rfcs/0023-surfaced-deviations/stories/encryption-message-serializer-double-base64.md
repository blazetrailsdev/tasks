## Context

Our encryption wire format is **not byte-compatible with Rails/MRI** because
header values get base64-encoded twice.

Mechanism: `packages/activerecord/src/encryption/message-serializer.ts`
`encodeIfNeeded` (~L124) base64-encodes **every string** it serializes —
payload _and_ header values. But `Aes256Gcm` (`encryption/cipher/aes256-gcm.ts`)
already stores binary header values (IV, auth tag) as base64 **strings** on the
`Message` headers, so the serializer re-encodes them: the JSON envelope carries
`base64(base64(iv))` where MRI carries `base64(iv)`. See the explanatory
comment in `encryption/encryptable-record.test.ts:607` ("double-base64 —
because Aes256Gcm.encrypt adds headers as already-…") and the prior sizing note
in the old trails tracker (`activerecord-100-plan.md` Batch 107).

Rails counterpart: `activerecord/lib/active_record/encryption/message_serializer.rb`
(`encode_if_needed` base64-encodes only values flagged binary / non-UTF8;
headers hold raw bytes until serialization).

**This is architectural, not a one-line fix**: the clean fix is for `Aes256Gcm`
to store **raw bytes (Buffer)** in message headers and let `MessageSerializer`
do the single base64 hop, exactly like MRI. That changes the on-disk
ciphertext envelope, so it's a **breaking change for existing stored
ciphertexts** and must ship with a migration/compat path:

- Decryption must accept BOTH formats (try single-decode; fall back to
  double-decode) so existing rows keep decrypting — mirror how Rails handles
  `support_unencrypted_data` style downgrades with a config flag if needed.
- Encryption switches to the MRI format.
- `encrypted-fixtures` and any parity fixtures regenerate.

Value: cross-decryptability with real Rails apps (a ciphertext written by
Rails decrypts in trails and vice versa) — worth pinning with a hard-coded
MRI-produced ciphertext fixture in the test.
