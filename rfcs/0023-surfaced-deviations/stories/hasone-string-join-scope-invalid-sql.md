## Context

Rails supports `has_one` associations whose scope includes a raw SQL string join:

```ruby
has_one :very_special_comment_with_string_joins,
  -> { joins("JOIN posts AS p1 ON comments.post_id = p1.id").where.not(p1: { id: 999999 }) },
  class_name: "VerySpecialComment"
```

`Post.joins(:very_special_comment_with_string_joins).first` works in Rails.
In trails, the same produces invalid SQL on SQLite (`no such column: p1.id`)
because the alias `p1` introduced in the string join is not visible in the
subsequent WHERE clause as generated by the query builder. Surfaced when porting
`test_eager_association_with_scope_with_string_joins` in
`associations/join-model.test.ts` (wave 5, PR #3617) — test is currently
`it.skip`'d there.

trails source: `packages/activerecord/src/test-helpers/models/post.ts`
`verySpecialCommentWithStringJoins` hasOne definition.
