## Context

Surfaced while measuring api:compare deltas for PR #4867. Every
`pnpm api:compare` run leaves the tree dirty:

```text
 eslint/rails-callback-invocations.json | 45 +++++++++++++++++++++++++---------
 1 file changed, 34 insertions(+), 11 deletions(-)
```

**Verified it is format-only, not stale data and not nondeterminism:**

- Two consecutive runs produce byte-identical diffs (deterministic).
- The regenerated file is **semantically identical** to the committed one —
  `json.load(new) == json.load(HEAD version)` is `True`. Only array layout
  differs (`"create": ["x"]` re-expanded to three lines).

Root cause is already documented in `.prettierignore:14-17`, for a **sibling**
file:

```text
# Generated by scripts/build-rails-privates-manifest.ts (prelint); committed
# but emitted with plain JSON.stringify whose array layout differs from
# Prettier's line-width-based collapsing.
eslint/rails-deprecated-methods.json
```

`scripts/build-rails-privates-manifest.ts` writes three artifacts —
`rails-private-methods.json` (:53), `rails-deprecated-methods.json` (:336), and
`rails-callback-invocations.json` (:430) — but only the middle one is
prettier-ignored. So the same fix was applied to one of three, and
`rails-callback-invocations.json` still churns. (`rails-private-methods.json`
does not churn in practice; its layout happens to match Prettier's.)

Cost: every agent running api:compare gets a spurious dirty file, must remember
`git checkout -- eslint/`, and risks committing the reformat as diff noise or a
conflict against a sibling agent.

## Acceptance criteria

- [ ] `pnpm api:compare` leaves the tree clean (`git status --short` empty).
- [ ] Fix matches the established precedent — add
      `eslint/rails-callback-invocations.json` to `.prettierignore` alongside
      `eslint/rails-deprecated-methods.json` under the existing comment — or,
      if preferred, make the generator emit Prettier-formatted output for all
      three artifacts so no ignore entry is needed.
- [ ] Whichever is chosen, all three generator outputs are treated
      consistently, so the next added artifact doesn't reintroduce this.
