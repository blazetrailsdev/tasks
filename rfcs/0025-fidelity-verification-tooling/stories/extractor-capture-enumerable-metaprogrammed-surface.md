## Context

Follow-up to `extractor-capture-metaprogrammed-ruby-surface` (RFC 0025), which
taught `scripts/api-compare/extract-ruby-api.rb` to capture `class_attribute`/
`cattr_accessor`/`mattr_accessor`, the bare `alias` keyword, and `delegate …
to:` (including `delegate(*CONST, to:)` splat-expansion).

Two in-tool allow-sets remain in `scripts/api-compare/extra-surface.ts`:

- `RAILS_RELATION_VALUE_METHODS` — `limitValue`, `groupValues`, … generated by
  `Relation::VALUE_METHODS.each { … class_eval "def #{name}_value …" }` in
  `relation/query_methods.rb:162`. The `_value`/`_values`/`_clause` suffix is
  chosen by a `case name when *MULTI_VALUE_METHODS …` over constants defined in
  a DIFFERENT file (`relation.rb:59-65`).
- `RAILS_DSL_GENERATED` — `integer`, `string`, `json`, … generated by
  `define_column_methods` in `connection_adapters/**`.

Both are enumerable `class_eval`/`define_method` loops driven by symbol-array
constants, often cross-file. The static extractor models neither, so the TS
ports look novel and the allow-sets paper over them in-tool. Removing the
allow-sets requires the extractor to model these loops (cross-file constant
resolution + per-element suffix derivation).

## Acceptance criteria

- `extract-ruby-api.rb` emits the `Relation::VALUE_METHODS.each` accessors
  (`*_value`/`*_values`/`*_clause`) and `define_column_methods` column DSL
  methods, resolving the driving symbol-array constants (incl. cross-file
  `relation.rb` → `query_methods.rb`).
- `RAILS_RELATION_VALUE_METHODS` and `RAILS_DSL_GENERATED` are deleted from
  `extra-surface.ts`; `pnpm api:extra --package activerecord --novel-only`
  shows no regression (those names stay non-novel via the manifest).
- `pnpm api:compare` gate + the api-compare test suites still pass.
