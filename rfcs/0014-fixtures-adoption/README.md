# RFC 0014 — Fixtures adoption: opportunistic Tier-1 conversion, defer the sweep

## Summary

The canonical-fixtures test infrastructure shipped (PR #2755 et al.) and is a
deliberate mirror of Rails' fixtures model. The question this RFC settles is
**how far to push adoption across the existing AR test suite.** The generated
adoption inventory (`pnpm fixtures:adoption:inventory`) is unambiguous: **defer
Phase G as a sweep.** Only a handful of files are mechanically convertible; the
suite is overwhelmingly bespoke-per-describe schemas and no-DB-op unit tests
that a canonical-fixture loader cannot serve. This RFC records that decision,
captures the one small piece of pickable work (opportunistic Tier-1 conversion,
done the Rails way), and makes the broad sweep an explicit **non-goal**.

## Do as Rails does

The conversion target is **Rails' fixtures shape**, and our infra already mirrors
it 1:1 — converting a file means moving it onto that shape, not inventing one:

| Rails (`activerecord/lib/active_record/…`)                                | trails mirror                                                                                                                                              |
| ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `test_fixtures.rb` — `fixtures :name` macro + the `name(:label)` accessor | `useFixtures` (`test-helpers/use-fixtures.ts:323`)                                                                                                         |
| `test_fixtures.rb` — `use_transactional_tests = true` (default)           | `useTransactionalTests` (`test-helpers/use-transactional-tests.ts:10`, comment cites Rails)                                                                |
| class-level `fixtures` + transactional wrap                               | `useHandlerFixtures` (`test-helpers/use-handler-fixtures.ts:45`)                                                                                           |
| `fixtures.rb` — `ActiveRecord::FixtureSet`, `test/fixtures/*.yml`         | the fixtures registry (`test-helpers/fixtures-registry.ts`)                                                                                                |
| canonical shared schema (Rails `schema.rb`, loaded once)                  | `TEST_SCHEMA` (`test-helpers/test-schema.ts`) seeded once via `globalSetup` (`template-global-setup.ts`) + `seedSchemaSignatures` (`define-schema.ts:487`) |

So a Tier-1 conversion = replace a file's bespoke inline `class X extends Base` +
`beforeAll(defineSchema)` with **canonical models + `fixtures :name` registry
entries + `name(:label)` lookups** — exactly Rails' `test/cases` style. The gold
standard is PR #2766 (PessimisticLocking) / #2788 (sqlite3 explain → canonical
Author + fixtures); the shallow form to avoid is #2764 (boilerplate trim only).

## Motivation

`fixtures-adoption-inventory.md` (script-generated by
`scripts/fixtures-inventory/inventory.ts`, wired at `package.json:24`; do not
hand-edit) classifies all 497 AR test files:

| Tier | Meaning                           | Count | %     |
| ---- | --------------------------------- | ----- | ----- |
| 1    | auto-eligible (mechanical)        | 9     | 1.8%  |
| 2    | loader-gap blocker                | 0     | 0.0%  |
| 3    | Phase-6 hazard / bespoke schema   | 146   | 29.4% |
| 4    | no DB ops / intentional isolation | 342   | 68.8% |

Of the 9 Tier-1 files, **4** already call `useFixtures`; **5** are unconverted.
The inventory's recommendation: _"do NOT spin up a 12–18 PR sweep. Convert the
small Tier-1 pool opportunistically (bundled into adjacent PRs touching those
files) and treat fixtures adoption as a per-file Rails-parity nicety, not a
program."_ The binding constraint was never the (now-shipped) transactional
infra — it's that ~98% of the suite is bespoke models or no-DB units that
canonical fixtures don't fit. A blind sweep yields ~8% (the canary-era reality).

> Counts are as-of-generation; several Tier-1/adjacent files have since been
> ported in unrelated canonical-migration PRs (e.g. locking, dirty, errors).
> **Re-run `pnpm fixtures:adoption:inventory` for the current pool before
> picking up the story below.**

## Design

**Stance: opportunistic, not programmatic.**

- Convert the unconverted **Tier-1** files when a PR is already touching them
  (the [tier1-opportunistic-conversion](stories/tier1-opportunistic-conversion.md)
  story), onto the Rails fixtures shape above — not a shallow
  `beforeAll(defineSchema)` trim.
- Do **not** schedule a Tier-3/Tier-4 sweep — see §Non-goals.

The inventory itself is a **generated artifact**
(`scripts/fixtures-inventory/inventory.ts` → `docs/activerecord/fixtures-adoption-inventory.md`).
It is not hand-maintained tracking; it's a planning report regenerated on
demand. That has a consequence for the cutover — see §Open questions.

## Alternatives considered

- **Run the full Phase-G sweep (12–18 PRs).** Rejected by the inventory's data —
  ~8% yield; the bespoke/no-DB tail isn't loader-serviceable.
- **Adopt the untracked perf-driven re-framing.** Two untracked local drafts
  (`defineschema-to-fixtures-migration.md`, `fixtures-migration-backlog.md`)
  re-open the defer decision, ordered by killing table-drops for performance.
  **Out of scope here** (treated as local scratch, not repo content). If that
  performance angle is promoted, it warrants its **own** RFC — it's a distinct
  thesis (perf, not Rails-parity) from this one.

## Rollout

1. [tier1-opportunistic-conversion](stories/tier1-opportunistic-conversion.md) —
   low priority; pick up only alongside PRs already in those files.

## Non-goals (explicit — do not lift into the ready queue)

- **Tier-3 (146 bespoke per-describe schemas).** Each needs bespoke surgery; a
  canonical loader can't serve them. Not a program.
- **Tier-4 (342 no-DB-op / intentional-isolation files).** Nothing to convert —
  Rails wouldn't fixture these either (they exercise SQL generation / in-memory
  behavior, not persisted rows).
- **A scheduled multi-PR adoption sweep.** Per the inventory recommendation.

## Open questions

1. **The inventory doc is script-generated into `docs/`.** When `decommission-docs`
   deletes `docs/activerecord/fixtures-adoption-inventory.md`, the next
   `pnpm fixtures:adoption:inventory` run regenerates it — which the Phase-4
   drift guard would then flag. Resolve one of: (a) retarget the script's output
   out of `docs/` (e.g. a gitignored `.fixtures/` or stdout-only), (b) allowlist
   the path like `parity-verification.md`, or (c) drop the committed copy and
   keep the script as an on-demand report. **Recommend (a)** — a generated report
   shouldn't live in the tracked plan tree; `decommission-docs` should land the
   script-output retarget in the same change that deletes the committed copy.
